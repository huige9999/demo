字节流消费策略

到达是同一回事，消费是另一回事。把同一根 TCP 字节流，交给不同的消费者，看「第一次可用产出」何时出现。

流开始 第一次可用产出 流结束

实验 0 · 控制实验：证明「只有一种到达方式」

const res = await fetch('/same-bytes?ct=' + ct);
console.log(res.headers.get('content-type')); // 头一到这里就拿到了
for await (const chunk of res.body) {          // 你自己在读流
  logChunk(chunk);                              // 节奏完全由服务端节流决定
}

预期：切 Content-Type，chunk 的到达节奏、块大小、块数量【完全一样】，变的只有那个 header 字符串。

实验 1 · 缓冲式消费（.blob() 路线）

const chunks = [];
for await (const chunk of res.body) {
  chunks.push(chunk);              // 攒着……什么都不干
}
// —— 流到这里才结束 ——
const blob = new Blob(chunks);
video.src = URL.createObjectURL(blob); // 此刻才能播

预期：流结束【之前】<video> 一直是黑的；只有进度条在涨。流一结束，视频立刻可播 —— 「第一次可用产出」贴着「流结束」。

实验 2 · 增量解析式消费（NDJSON 路线）

let buffer = '';
for await (const chunk of res.body) {
  buffer += dec.decode(chunk);
  let idx;
  while ((idx = buffer.indexOf('\n')) !== -1) {  // 边读边切行
    const line = buffer.slice(0, idx);
    buffer = buffer.slice(idx + 1);
    appendRow(JSON.parse(line));                 // 切出一行就立刻 emit
  }
  // buffer 里残留的「半行」留到下个 chunk 拼上再切
}

预期：表格里的行一条一条蹦出来，远在流结束之前 —— 「第一次可用产出」很靠左。
注意：一行 JSON 可能被拆在两个 chunk 里，buffer 必须保留半行。

实验 3a · 增量渲染式：浏览器内置消费者（<video src>）

// 不写一行 JS —— 把流直接交给浏览器
<video src="/raw?speed=mid" controls></video>
// 浏览器自己发 Range 请求、自己解码、自己消费字节流，边下边播。
// 打开 DevTools → Network，能看到 206 Partial Content。

预期：视频在字节还没下完时【就开始播了】（边下边播）。
要讲清楚：边下边播靠的是 Range 请求 + 浏览器自带的解码消费者，不是 Content-Type: video/mp4 触发了某种「流式模式」。
