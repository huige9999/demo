# 字节流消费策略 · demo 实验方案

> 配套文档：[`0001-Http响应体第一性原理.md`](./0001-Http响应体第一性原理.md)
> 本方案只做**设计**，不含实现代码。先把"要让你摸到什么"和"怎么摸"
> 想清楚，再动手。

---

## 一、这个 demo 要让你"摸到"的两件事

把第一性原理那篇文档的核心结论拆成**两个可被肉眼验证的命题**，
每个命题配一组实验：

| 命题 | 怎么验证 | 对应实验 |
|---|---|---|
| **① 字节流只有一种到达方式** | 用**同一段** `for await (chunk of res.body)` 代码去接不同的接口，观察"收 chunk 的节奏"完全一样 | 实验 0（控制实验）|
| **② 同一份字节，消费策略不同，结果天差地别** | 拿到 chunk 之后分别走"攒满再用 / 边读边解析 / 边读边喂解码器"三条路，观察三者"第一次出现可用产出"的时间点 | 实验 1 / 2 / 3 |

一句话：**到达是同一回事，消费是另一回事。** 这个 demo 就是要把"另一回事
"三个字变成你能在屏幕上看到的东西。

## 二、整体架构

### 技术选型：Node 原生 `http` + 原生 HTML/JS，零依赖

理由（和"第一性原理"的主题必须自洽）：

- 这个 repo 本来就是 Node 学习仓库，用原生 `http` 最贴气质；
- **零依赖 = 没有魔法**。服务端就是 `fs.createReadStream` + `res.write`，
  客户端就是 `fetch` + `for await`，每一行都能对着文档讲清楚；
- 不引入框架/打包器，避免"是框架帮我做的吗"这种干扰项。

### 架构图

```
┌──────────────────────────────────────────────────────────────┐
│  Node http 服务端  (唯一职责：把字节按节奏写到 socket)        │
│  ──────────────────────────────────────────────────────────  │
│   /same-bytes?ct=...   Content-Type 由参数定   节流   ←实验0  │
│   /api/blob            application/octet-stream 节流  ←实验1  │
│   /api/ndjson          application/x-ndjson    节流  ←实验2  │
│   /raw                 video/mp4   支持 Range   节流  ←实验3a │
│   /fmp4                video/mp4   分片 fMP4     节流  ←实验3b │
│   /                    静态返回 index.html                    │
└──────────────────────────────────────────────────────────────┘
                         │  ← 同一根 TCP 字节流（真相就在这根线上）
                         ▼
┌──────────────────────────────────────────────────────────────┐
│  浏览器 index.html   （每张实验卡片里都跑这一段）             │
│     for await (const chunk of res.body) { 收到一块 chunk }   │
│                                                               │
│  差别只在 chunk 拿到之后交给谁：                              │
│   卡片0 → 只 log（证明到达方式相同）                          │
│   卡片1 → 攒进数组，流结束才 createObjectURL（缓冲式）        │
│   卡片2 → 按 \n 切分，每行立刻 emit 一条记录（增量解析式）    │
│   卡片3a → 交给 <video src>，浏览器内置消费者（增量渲染式）   │
│   卡片3b → sourceBuffer.appendBuffer 喂 <video>（你当消费者） │
└──────────────────────────────────────────────────────────────┘
```

## 三、三个必须先想清楚的设计决策

### 决策 1：服务端必须"故意变慢"（demo 的命根子）

localhost 上 536KB 一次 `data` 事件可能就发完了，你**根本看不见流**。
所以服务端不能直接 `stream.pipe(res)`，而要**手动节流**：读一块、写一块、
`pause()`、`setTimeout` 后再 `resume()`，人为制造"一滴一滴"的效果。

> 这是整个 demo 能不能"摸到"的关键。不节流，所有实验都会退化成"瞬间
> 完成"，缓冲式和增量式看起来就没区别了——那这个 demo 就白做了。

节流做成**可调速度**（客户端有个速度滑块，通过 query 参数告诉服务端每块
间隔多少 ms），这样你可以亲手把速度拉慢，把"流"看清楚。

### 决策 2：服务端也要"裸"，证明它没干别的

服务端的实现要刻意保持" dumb "：它**只负责把 `video.mp4` 的字节按节流
节奏写出去**，不解析、不组装、不挑客户端。每个接口的差别只是
`Content-Type` 头不同 + 字节来源不同（mp4 文件 / 服务端生成的文本），
**处理逻辑完全一样**。这正好坐实文档里那句"网络层只干一件事：把字节按
顺序可靠地送到"。

### 决策 3：实验 0 是地基，必须最先做、最有说服力

实验 0（控制实验）是整个 demo 的逻辑基石。它直接证明文档"第 4 层"的
论断：**`Content-Type` 只是个标签，不改变字节到达方式**。这个实验跑通
了，后面三个消费策略实验才有支点。

## 四、实验清单

> 每个实验 = 首页上的一张可独立运行的"卡片"。卡片之间互不干扰，可重复
> 跑、可调速。

### 实验 0 · 控制实验：证明"只有一种到达方式"

- **接口**：`GET /same-bytes?ct=<video/mp4 | application/json | application/octet-stream | text/event-stream>`
- **关键设计**：**同一份字节**（都是 `video.mp4` 的内容）、**同一种节流**，
  只改 `Content-Type` 这一个头。
- **客户端代码**（所有卡片共用这一段骨架）：

  ```js
  const res = await fetch(`/same-bytes?ct=${ct}`);
  console.log(res.headers.get("content-type")); // 头一到这里就拿到了
  for await (const chunk of res.body) {          // 你自己在读流
    logChunk(chunk);                              // 节奏完全由服务端节流决定
  }
  ```

- **预期看到**：不管 `ct` 是哪种，chunk 的**到达节奏、块大小、块数量都一
  模一样**。变的只有 `res.headers.get("content-type")` 那个字符串。
- **结论**：这就是文档说的——把它换成 SSE、换成 mp4，**这一段代码的行为
  一模一样**。头不决定行为，字节流只有一种到达方式。

### 实验 1 · 缓冲式消费（`.blob()` 路线）

- **接口**：`GET /api/blob`（`application/octet-stream`，节流，字节来自 mp4）
- **客户端**：`fetch` → 把每个 chunk 攒进数组 → 流结束后拼成 `Blob` →
  `URL.createObjectURL(blob)` → 赋给 `<video>`。
- **预期看到**：在流结束**之前**，`<video>` 是黑的、不能播；只有进度条
  在涨。**流一结束**，视频立刻可播。
- **时间线标记**：`第一次可用产出` 这个点 = `流结束` 这个点（两者重合）。
- **对应文档**：缓冲式姿态——`.json()` / `.text()` / `.blob()`，必须读
  到流结束才能动手。

### 实验 2 · 增量解析式消费（NDJSON 路线）

- **接口**：`GET /api/ndjson`（`application/x-ndjson`，节流，**字节来自服务端
  生成的文本**——每行一个 JSON 对象，例如模拟的"学生记录"）
- **客户端**：`fetch` → 维护一个跨 chunk 的字符串缓冲 → 每收到 chunk 就
  按 `\n` 切分，**切出完整一行的立刻 `JSON.parse` 并往页面上 append 一行**。
- **预期看到**：表格里的行**一条一条蹦出来**，远在流结束之前页面就已经
  有内容了。
- **时间线标记**：`第一次可用产出` 出现得很早（第一块 chunk 切出第一行时）。
- **对应文档**：增量解析式姿态——`EventSource`、NDJSON、流式 JSON 解析器，
  边读边按协议切分，绝不等到流结束。
- **注意**：这里**不能用 mp4 字节**（二进制当 JSON 解析会报错），所以这个
  接口的字节来源是服务端现造的文本。这也是合理的——它证明"增量解析"这
  种策略对**任何**字节流都成立。

### 实验 3 · 增量渲染式消费（video 路线，分两个层次）

这一组对应文档 Q4，最值得做，也最有视觉冲击。分两小题：

#### 实验 3a · 浏览器内置消费者（默认做这个）

- **接口**：`GET /raw`（`video/mp4`，**支持 Range**，节流）
- **客户端**：直接 `<video src="/raw" controls>`，**不写一行 JS**。
- **预期看到**：视频在字节还没下完时**就开始播了**（边下边播）。
- **要讲清楚的点**：边下边播靠的是 **Range 请求 + 浏览器自带的解码消费者
  **，**不是** `Content-Type: video/mp4` 这个头触发了某种"流式模式"。打开
  DevTools Network 面板能看到 `206 Partial Content`。
- **对应文档**："浏览器为少数场景内置了专门的解释器 + 消费者，看起来像
  自动，本质还是上层在消费同一个字节流。"
- **优点**：用准备好的 `video.mp4` 直接就能跑，零额外处理。

#### 实验 3b · 你自己当消费者：MSE（进阶，需先转 fMP4）

- **接口**：`GET /fmp4`（`video/mp4`，节流；服务端返回** fragmented MP4**）
- **客户端**：`fetch` → `for await (chunk)` → 每块 `sourceBuffer.appendBuffer(chunk)`
  喂给 `MediaSource` → `<video>` 边收边播。
- **意义**：这一题把"消费者"从浏览器手里**夺回到你自己手里**——你亲自
  读流、亲自喂解码器，彻底坐实"上层代码决定怎么消费"。
- **前置条件（重要）**：MSE 的 `SourceBuffer` 只吃 **fragmented MP4**，准备
  好的 `video.mp4` 多半是**非分片**的，直接喂会报错。需要先用 ffmpeg 转一次：

  ```bash
  ffmpeg -i assets/video.mp4 \
    -movflags empty_moov+default_base_moof+frag_keyframe+separate_moof \
    -frag_size 100000 assets/video-fmp4.mp4
  ```

  > 如果环境没装 ffmpeg，**实验 3b 先不做**，3a 已经足够说明问题。3b 作为
  > "想再深一层"的进阶项保留。

### 5.2 时间线可视化（全 demo 的"点睛之笔"）

每张卡片底部画**同一根横向时间轴**，上面标三个点：

```
流开始 ●───────────●───────────────────────● 流结束
                 ↑                        ↑
            "第一次可用产出"          (流到此结束)
```

把实验 1 / 2 / 3 的时间线**并排放**，立刻能看到：

| 实验 | "第一次可用产出"落在哪 |
|---|---|
| 实验 1（缓冲式） | **贴着"流结束"** —— 攒满才能用 |
| 实验 2（增量解析） | **很靠左** —— 第一块就出活 |
| 实验 3（增量渲染） | **很靠左** —— 边收边播 |

> 这一张并排对比图，就是把文档那张"三种消费姿态"表格**变成画面**。看完
> 不需要再解释。

### 5.3 控件

- **开始**：触发该卡片的 `fetch`（每张卡片独立，互不干扰）；
- **速度**：调节服务端节流间隔（慢/中/快），慢档用来把"流"看清楚；
- **重置**：清空日志、进度、产出区，方便反复跑。

## 六、服务端接口清单（一张表定下来）

| 路径 | Content-Type | 支持 Range | 节流 | 字节来源 | 服务于 |
|---|---|---|---|---|---|
| `GET /` | text/html | — | 否 | index.html | 静态首页 |
| `GET /same-bytes?ct=&speed=` | 由 `ct` 参数决定 | 否 | 是 | video.mp4 | 实验 0 |
| `GET /api/blob?speed=` | application/octet-stream | 否 | 是 | video.mp4 | 实验 1 |
| `GET /api/ndjson?speed=` | application/x-ndjson | 否 | 是 | 服务端生成文本 | 实验 2 |
| `GET /raw` (+`Range`) | video/mp4 | **是** | 是 | video.mp4 | 实验 3a |
| `GET /fmp4?speed=` | video/mp4 | 否 | 是 | video-fmp4.mp4 | 实验 3b（进阶）|

> 服务端实现要点（设计层面，不写代码）：节流用「读一块 → `res.write` →
> `pause` → `setTimeout(speed) → resume`」；Range 用 `fs.createReadStream`
> 的 `{start,end}` + 回 `206`/`Content-Range`；NDJSON 用一个生成器一边
> 造文本行一边按节流写。

## 七、技术风险与坑（动手前必须知道）

1. **localhost 太快 → 必须节流**（决策 1）。这是头号坑，不处理 demo 直接
   失效。
2. **MSE 只认 fragmented MP4**（实验 3b）。准备好的视频多半是非分片的，
   要么用上面给的 ffmpeg 命令转一次，要么 3b 暂缓。
3. **实验 3a 必须实现 Range**，否则 `<video src>` 拖进度条 / 边下边播会
   异常。Range 不难，但不能漏。
4. **NDJSON 跨 chunk 切分**：一个 JSON 行可能被拆在两个 chunk 里，客户端
   必须保留"半行"缓冲到下一个 chunk 拼上再切。这个细节正好是"增量解析
   式"的真实复杂度，值得在卡片里点出来。
5. **浏览器对 `video/mp4` 的内置消费者会"接管"流**（实验 3a）：这恰恰是
   文档 Layer 4 要讲的点，不是坑，是要重点演示的现象——别误以为是 bug。

## 九、建议的实施顺序

1. **先搭骨架**：`server.js` 起服务、静态返回 `index.html`、做一个能节流
   的 `/same-bytes`。
2. **实验 0 先跑通**：能看见不同 `ct` 下 chunk 节奏一致 → 地基立住。
3. **再做时间线 UI 组件**：这是后面所有实验共用的可视化，先做好。
4. **依次填实验 1 → 2 → 3a**：每填一个，时间线上多一根条，对比逐渐成型。
5. **最后（可选）实验 3b**：转 fMP4 + MSE，作为进阶收尾。

## 十、待你拍板的几个点

在动手写代码前，下面几项建议先定一下（都有我的推荐默认值）：

1. **技术栈**：推荐 **Node 原生 `http` + 原生 HTML/JS，零依赖**（理由见第
   二节）。是否同意？如果想用 Express 也可以，但会多一层"是 Express 帮的
   吗"的噪声。
2. **实验 3b（MSE）做不做**：推荐 **默认不做**，先把 0/1/2/3a 做扎实；3b
   作为进阶。需要的话你得确认本机有 ffmpeg。
3. **节流速度档位**：推荐三档（慢 ~200ms/块 / 中 ~50ms / 快 ~10ms），慢档
   用于演示。可调。
4. **首页交互形式**：推荐**一张长页 + 若干纵向卡片 + 并排时间线对比**（第
   五节）。如果你更想要"一次只看一个实验、全屏切换"的形式，也可以。

> 以上 4 点确认后（或直接采纳默认值），即可进入实现阶段。
