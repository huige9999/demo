# SSE 响应 vs 普通流式响应

> 配套 demo：[`0003-demo/`](../0003-demo/)  
> 前置阅读：[`0001-Http响应体第一性原理.md`](./0001-Http响应体第一性原理.md)

---

## 一、先对齐概念：它们都是「流」，差别在「协议 + 消费者」

从 TCP 层看，**SSE 和普通流式响应没有区别**——都是 HTTP body 里一串按顺序到达的字节。

差别出在两层：

| 层次 | SSE | 普通流式响应 |
|---|---|---|
| **服务端写法** | 固定文本格式 + 特定响应头 | 任意格式，自己约定 |
| **客户端消费** | `EventSource`（浏览器内置解析器） | `fetch` + 读 `ReadableStream`，自己解析 |

一句话：**SSE 是「带标准文本协议的流」；普通流式响应是「裸字节流 + 自定义协议（或无协议）」。**

---

## 二、SSE 是什么

### 2.1 服务端必须做的事

1. 响应头：`Content-Type: text/event-stream`
2. **故意让响应长时间不结束**（或按需结束）
3. 按 SSE 文本协议写 body，每条消息以 `\n\n` 结尾

最小消息格式：

```
data: 你好\n\n
```

带事件名、id、多行 data：

```
event: token
id: 42
data: 第一行
data: 第二行

```

常用响应头：

```
Content-Type: text/event-stream; charset=utf-8
Cache-Control: no-cache
Connection: keep-alive
```

> 注意：Node 原生 `http` 里**不要**手动设 `Transfer-Encoding: chunked`，交给运行时即可。

### 2.2 后端示例（Node 原生 http）

```js
function handleSse(req, res) {
  res.writeHead(200, {
    "Content-Type": "text/event-stream; charset=utf-8",
    "Cache-Control": "no-cache",
    "Connection": "keep-alive",
  });

  const tokens = ["你", "好", "，", "这", "是", "SSE"];
  let i = 0;

  const timer = setInterval(() => {
    if (i >= tokens.length) {
      // 可选：发一条结束事件后关闭
      res.write("event: done\ndata: [DONE]\n\n");
      clearInterval(timer);
      res.end();
      return;
    }
    // 每条 SSE 消息：data 行 + 空行
    res.write(`data: ${JSON.stringify({ token: tokens[i], index: i })}\n\n`);
    i++;
  }, 300);

  req.on("close", () => clearInterval(timer));
}
```

要点：

- 用 `res.write()` 一段段写，**不要等全部生成完再 `res.end()`**
- 每条消息必须以 `\n\n` 结束
- 客户端断开时清理定时器（`req.on("close")`）

### 2.3 前端示例（EventSource）

```js
const es = new EventSource("/api/sse");

es.onmessage = (e) => {
  const payload = JSON.parse(e.data);
  appendToken(payload.token);
};

es.addEventListener("done", () => {
  es.close();
});

es.onerror = () => {
  // EventSource 会自动重连；若服务端已 end()，这里会反复触发
  es.close();
};
```

要点：

- **只能用 GET**，不能带自定义 body
- **不能设自定义请求头**（Cookie 同源会自动带）
- 浏览器帮你：按 `\n\n` 切消息、解析 `data:` / `event:` / `id:`、断线重连
- `onmessage` 对应没有 `event:` 字段的默认消息；具名事件用 `addEventListener("done", ...)`

## 三、普通流式响应是什么

### 3.1 服务端：只负责「持续写 bytes」

没有标准格式。常见几种：

| 风格 | Content-Type | body 形态 | 典型场景 |
|---|---|---|---|
| 裸文本流 | `text/plain; charset=utf-8` | 直接写 UTF-8 字符串 | LLM token 流、日志 tail |
| NDJSON | `application/x-ndjson` | 每行一个 JSON + `\n` | 结构化增量数据 |
| 二进制流 | `application/octet-stream` | 任意 bytes | 文件分块下载 |

示例（裸文本流，模拟 token 输出）：

```js
function handlePlainStream(req, res) {
  res.writeHead(200, {
    "Content-Type": "text/plain; charset=utf-8",
    "Cache-Control": "no-cache",
  });

  const tokens = ["你", "好", "，", "这", "是", "普通流"];
  let i = 0;

  const timer = setInterval(() => {
    if (i >= tokens.length) {
      clearInterval(timer);
      res.end();
      return;
    }
    res.write(tokens[i]); // 没有 data: 前缀，没有 \n\n 帧边界
    i++;
  }, 300);

  req.on("close", () => clearInterval(timer));
}
```

和 SSE 的对比：**同样是 `setInterval + res.write`，SSE 多了一层文本帧协议。**

### 3.2 前端：fetch + ReadableStream

```js
const res = await fetch("/api/stream");
const reader = res.body.getReader();
const dec = new TextDecoder();
let text = "";

while (true) {
  const { value, done } = await reader.read();
  if (done) break;
  text += dec.decode(value, { stream: true });
  output.textContent = text; // 每收到一块就更新 UI
}
```

或用 async iterator（更简洁）：

```js
const res = await fetch("/api/stream");
const dec = new TextDecoder();
let text = "";

for await (const chunk of res.body) {
  text += dec.decode(chunk, { stream: true });
  output.textContent = text;
}
```

要点：

- **fetch 不会因为 Content-Type 自动帮你切分**——chunk 边界由 TCP/HTTP 分块决定，和「语义单元」无关
- 若协议是「按行切 JSON」，需要自己维护跨 chunk 的 buffer（见 [`0002-demo`](../0002-demo/) 实验 2）
- 支持 POST、自定义 headers、AbortController 取消

## 四、核心对比表

| 维度 | SSE | 普通流式响应 |
|---|---|---|
| Content-Type | `text/event-stream` | 任意（`text/plain`、`application/x-ndjson`…） |
| body 格式 | 标准文本帧（`data:` / `event:` / `id:`） | 自定义或无格式 |
| 客户端 API | `EventSource` | `fetch` + `ReadableStream` |
| 请求方法 | 仅 GET | GET / POST / … 均可 |
| 自定义请求头 | 不支持 | 支持 |
| 取消请求 | `es.close()` | `AbortController` |
| 断线重连 | 浏览器内置 | 需自己实现 |
| 跨 chunk 解析 | EventSource 帮你做 | 自己维护 buffer |
| 方向 | 单向（服务端 → 客户端） | 通常单向；POST 流式请求体是另一话题 |
| 二进制 | 不适合（文本协议） | 可以 |

## 五、什么时候用哪个

**优先 SSE，当：**

- 只需要服务端单向推送（通知、进度、简单 token 流）
- 想要浏览器内置解析 + 自动重连
- GET 长连接足够

**用普通流式响应，当：**

- 需要 POST（例如带 prompt 的 LLM 请求）
- 需要自定义 Authorization 等请求头
- 协议不是 SSE 文本帧（NDJSON、protobuf、自定义分隔符）
- 需要传二进制
- 需要精细控制取消、背压、多路复用

**反例（常见误区）：**

- 「看到 `text/event-stream` 头，fetch 就会自动按事件切分」→ **错**，必须自己解析或用 EventSource
- 「SSE 和普通流底层不一样」→ **错**，底层都是同一根字节流（见 0001 文档）

## 六、用裸 fetch 读 SSE（理解 EventSource 在干什么）

EventSource 本质上就是在做这件事：

```js
const res = await fetch("/api/sse");
const dec = new TextDecoder();
let buf = "";

for await (const chunk of res.body) {
  buf += dec.decode(chunk, { stream: true });
  let sep;
  while ((sep = buf.indexOf("\n\n")) !== -1) {
    const frame = buf.slice(0, sep);
    buf = buf.slice(sep + 2);
    // 解析 frame 里的 data: / event: 行……
    const dataLine = frame.split("\n").find((l) => l.startsWith("data: "));
    if (dataLine) {
      const data = dataLine.slice(6);
      appendToken(JSON.parse(data).token);
    }
  }
}
```

这和 NDJSON「按 `\n` 切行」是同一类 **增量解析式消费**——只是分隔符和字段规则不同。

## 七、demo 说明

[`0003-demo/`](../0003-demo/) 提供两个并排卡片，用**同一段 token 序列**演示：

| 接口 | 类型 | 客户端 |
|---|---|---|
| `GET /api/sse?speed=` | SSE | `EventSource` |
| `GET /api/stream?speed=` | 普通文本流 | `fetch` + `for await` |

启动：

```bash
node 0003-demo/server/server.js
# 浏览器打开 http://localhost:3001/
```

可调速度（slow / mid / fast），观察：

- SSE：每条 `data:` 消息触发一次 `onmessage`
- 普通流：每次 TCP chunk 到达触发一次循环体（chunk 边界 ≠ token 边界）
