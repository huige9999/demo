## 一、这个站的「三层防御」，一眼看穿

抓 `huanmengacg.com` 的最大错觉是：**以为拿到 HTML 就等于拿到正文**。
实际拿到的是一份**被打乱、被掺沙子**的 HTML，必须解码才能读。它有三层：

| 层 | 它干了什么 | 看起来像什么 | 解法 |
|---|---|---|---|
| ① Cloudflare 人机校验 | 拦掉「不像浏览器」的请求 | 默认 UA 返回 `Just a moment...` | 带**真实浏览器请求头** |
| ② 段落顺序打乱 | 正文 `<p>` 在源码里是**乱序**的 | 看第一段可能跳到结局 | 解析 `order`，**按数字升序排序** |
| ③ 诱饵段落 | 混入若干**拼接的半句话** | 混进正文读起来像胡言乱语 | 带 `class="<data-c>"` 的 `<p>` **丢弃** |

一句话：**这站不是「不给数据」，是「给脏数据，看你识不识货」。**

## 二、URL 结构（这个站抓任何书都通用）

```
https://www.huanmengacg.com/index.php/book/info/<bookId>        ← 书的目录页
https://www.huanmengacg.com/index.php/book_read_<bookId>_<chId>.html  ← 单章正文
```

- `bookId`：书的 id，在目录页 URL 里。例：Re0 是 `8768`。
- `chId`：**每章独立的 id**，从目录页的 `<ul id="all-chapters-raw">` 里
  的 `<a href>` 解析得到。

### ⚠️ 最大的坑：`chId` 不是「阅读顺序」

目录里 `chId` 和「显示序号」**不严格递增对应**。Re0 第二章里就有：

```
id=2547 → 第二章34
id=2546 → 第二章35      ← id 更小，序号反而更大！
id=2560 → 第二章47
id=2559 → 第二章48      ← 同样的「倒挂」
```

**结论：正确的阅读顺序 = 目录里 `<a>` 标签出现的先后顺序，绝对不能按 id 排序。**
在 `crawler/chapters.js` 里我把它写成**数组**（下标即顺序），从根上堵死这个坑。

> 抓别的书时，照搬这套：先从目录页抓出「`(chId, 显示序号)` 列表」，
> 按出现顺序存成数组，再逐个抓正文。

## 三、目录页怎么抓（通用步骤）

1. **请求目录页** `book/info/<bookId>`，带第二节那组浏览器头。
2. **定位章节列表**：`<ul id="all-chapters-raw">`，里面每个
   `<li name="box"><a href="...book_read_<bookId>_<chId>.html">序号+标题</a>`。
3. **正则抠 `(chId, 文本)`**：
   ```js
   const re = /book_read_\d+_(\d+)\.html"[^>]*>([^<]+)</g;
   ```
   `RegExp` 的**全局匹配按出现顺序**返回，天然就是阅读顺序。
4. 注意有些 `<li>` 带 `style="display:none;"`（隐藏的后续章节）——
   它们**仍然要抓**，只是默认折叠了，别因为 `display:none` 就跳过。

## 四、正文页的两层混淆 —— 核心解码逻辑

### 4.1 先看 #content 容器长什么样

```html
<div id="content"
     data-r="1966941106"
     data-t="38"
     data-c="r0fe8e29"      ← ★ 诱饵标记，每次刷新都变
     data-a="data-oyema"
     data-k="63979"
     style="display:flex;flex-direction:column;">
  <p style="order:1001" data-oyema="..." ...>睁开眼皮……</p>
  <p style="order:1000" ...>对昴来说，睡醒时的不适感……</p>   ← 源码里在后面，但 order 更小
  <p style="order:1148" ... class="r0fe8e29">★拼凑的诱饵半句★</p>
  ...
</div>
```

关键信息全在两个地方：
- **`data-c`**（容器上）：诱饵段落的 class 名，本次是 `r0fe8e29`，**下次变**。
- **`order:N`**（每个 `<p>` 的 style 上）：段落的真实排序键。

### 4.2 三步解码（对应实现 `extractParagraphs()`）

```
┌──────────────────────────────────────────────────────────────┐
│  ① 截取 #content 内部的 HTML                                   │
│  ② 读出 data-c = "r0fe8e29"（诱饵标记）                        │
│  ③ 遍历 <p>：                                                  │
│       order ← style 里的 order:N                              │
│       text  ← 剥标签 + 解 HTML 实体                            │
│       是诱饵？ ← <p> 的属性里有没有 class="r0fe8e29"            │
│              ├─ 是 → 丢弃                                      │
│              └─ 否 → 收进 items                                │
│  ④ items.sort((a,b) => a.order - b.order)  ← 恢复阅读顺序      │
│  ⑤ 输出 text 数组                                             │
└──────────────────────────────────────────────────────────────┘
```

### 4.3 一组实测数字（id=2513 一节）

```
#content 内 <p> 总数：     141
真段落（无诱饵 class）：    133   order 落在 1000..1146 连续区间
诱饵段落：                   8   order 在 1148..1155
解码后首段：              「对昴来说，睡醒时的不适感是无缘的东西。」
解码后末段：              「……怎么会有这么让人打从心底火大的家伙。」  ← 接上下一节
```

注意：**`order` 的具体数值每次刷新都会重新生成**（下次可能从 950 开始、
诱饵可能 6 个也可能 12 个），所以代码**只认规则，不认具体数字**。

### 4.4 三个易踩的细节

1. **截 `#content` 内部别贪多**。正文里不会出现嵌套 `<div>`，
   所以「`#content` 标签之后、第一个 `</div>` 之前」就是正文范围。
   截多了会把页脚的章节导航 `<p>` 也当正文抓进来。
2. **剥标签后要解 HTML 实体**。正文里有 `&ldquo;`/`&rdquo;`/`&hellip;`/`&#数字;`
   等，不全解会出现「&#20197;」这种残留。`&amp;` 必须放最后解，否则
   `&quot;` 会被二次解码。见 `cleanText()`。
3. **「拼接半句话」是诱饵的指纹**。诱饵特征是「上一句开头 + 下一句结尾」
   生硬拼一起，读起来语义断裂。可以用这个特征**人眼抽检**解码是否干净——
   但不要写死关键字检测，因为正文里也可能合法出现同样的词
   （例如「慎重地转动门把打开门」既在诱饵里、也在正文里出现过）。

## 五、Cloudflare 怎么过（请求头）

默认 `curl`/`wget`/`fetch`（裸 UA）会被拦到：

```html
<title>Just a moment...</title>
... Performing a security check ...
```

**带一组像浏览器的头即可直通**，不需要解题、不需要浏览器自动化：

```js
const HEADERS = {
  "User-Agent":
    "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 " +
    "(KHTML, like Gecko) Chrome/124.0.0.0 Safari/537.36",
  Accept: "text/html,application/xhtml+xml,application/xml;q=0.9,*/*;q=0.8",
  "Accept-Language": "zh-CN,zh;q=0.9,en;q=0.8",
  Referer: "https://www.huanmengacg.com/index.php/book/info/<bookId>",
  Connection: "close",   // 见第六节，防假死
};
```

并在拿到 HTML 后**主动检测拦截页特征**，命中就当失败重试：

```js
if (/just a moment/i.test(html) || /performing a security check/i.test(html)) {
  throw new Error("blocked by Cloudflare");
}
```

> 如果哪天这套头失效（Cloudflare 升级了），下一步才是考虑
> `puppeteer`/`undici` 的 TLS 指纹，别一上来就上重武器。

## 六、服务端「假死」—— 这站最阴的特性

抓取过程中你会发现：**单请求有时 2~6s 正常返回，有时却建连成功后
永远不回字节**，只能等超时。实测同一 URL 连打 8 次：

```
#1 OK   3335ms     #5 OK   3224ms
#2 OK   5321ms     #6 OK   6016ms
#3 OK   2955ms     #7 OK   2081ms
#4 FAIL timeout     #8 FAIL timeout     ← 约 20% 概率「假死」
```

**关键经验：假死时立刻重试，基本都能成。** 它不是封 IP、不是限流，
更像服务端连接处理有 bug。所以正确姿势是：

- **单请求超时给短点**（8s），别干等 30s；
- **退避给极小值**（500ms + 一点抖动），靠次数堆，而不是越退越久；
- **重试次数给足**（6 次），扛住 ~20% 的假死率。

```js
// fetch-chapters.js 的 fetchWithRetry() 精神：
const TIMEOUT_MS = 8000;          // 短超时
const RETRY = 6;                  // 多次
const backoff = 500 + random*500; // 极小退避
```

同时记得**每次请求带 `Connection: close` 并 `socket.destroy()`**，
别让假死的连接留在池里拖慢后续请求。

> 反面教材：我第一版用默认 30s 超时 + 1.5s 退避，结果 50 节里
> 几乎每节都要先超时一次（浪费 30s），半小时才抓了几节，还把一节
> 永久判失败。改成上面的策略后，50 节 0 失败、约 10 分钟跑完。

## 七、礼貌抓取与断点续抓

| 做法 | 理由 |
|---|---|
| 逐节抓，节间 `DELAY_MS≈1s` | 不并行、不压垮人家服务器，也避免触发更严的风控 |
| 单节失败只跳过、不中断整本 | 局部网络抖动不该连累整本书 |
| 每节单独落一个 `<序号>.txt` | 断点续抓、单独看、单独补抓都方便 |
| 失败节在汇总 md 里留占位 | 事后一眼看出哪几节缺了 |
| `node crawler/fetch-chapters.js <chId>…` | 单独补抓指定节 |

## 八、一次新抓取的清单（抄这个就行）

要抓这个站的某本书/某几章，按这个顺序：

1. **打开目录页** `book/info/<bookId>`，确认要抓的范围。
2. **生成章节清单**：抓目录页 → 正则抠 `(chId, 序号)` 列表 →
   按出现顺序写进一个新的 `chapters.js`（或改现有的）。
   ⚠️ 顺序认 `<a>` 出现顺序，**不认 id 大小**。
3. **跑 `fetch-chapters.js`**，它会自动：带浏览器头 → 检测拦截 →
   短超时 + 即时重试扛假死 → 解码（去诱饵 + 按 order 排序）→ 落 txt + 汇总 md。
4. **抽检**：看第一节开头、最后一节结尾、以及任何「id 倒挂」处
   （相邻两节 id 顺序和序号相反）的衔接是否通顺。
5. **看是否有诱饵残留**：抽几段，人眼判断有没有「语义突然断裂的拼接句」。

## 九、决策记录（为什么这么干）

| 问题 | 备选 | 选了什么 | 理由 |
|---|---|---|---|
| HTTP 客户端 | axios/got/puppeteer | **Node 原生 `https`** | 零依赖，和本 repo「Node 学习」气质一致；当前反爬用请求头就够，用不到浏览器自动化 |
| 顺序依据 | 按 `chId` 排序 / 按 `<a>` 出现顺序 | **出现顺序** | id 和序号不严格对应，按 id 排会错位（见第二节坑） |
| 假死处理 | 长超时干等 / 短超时+多次重试 | **短超时+多次重试** | 假死是随机连接 bug，立刻重试就成；干等纯粹浪费时间 |
| 诱饵识别 | 写死关键字 / 认 `class="<data-c>"` | **认 `data-c`** | `data-c` 每次刷新都变，但「诱饵 = 带 data-c 的 class」这条规则稳定；关键字不可靠（正文也含） |
| 输出格式 | 纯 txt / 一份 md / 数据库 | **每节 txt + 汇总 md** | txt 方便单独看和补抓，md 方便整本读 |

## 十、踩坑一览（下次别再犯）

- ❌ 拿到 HTML 就当正文 → ✅ 必须过 `extractParagraphs()` 解码。
- ❌ 按 `chId` 数字排序定阅读顺序 → ✅ 按目录里 `<a>` 出现顺序。
- ❌ 用默认 `req.timeout` 一个定时器 → ✅ 用「读不完也算超时」的兜底定时器
  + 出口必 `socket.destroy()`，否则假死连接堆积。
- ❌ `&amp;` 先于其它实体解码 → ✅ `&amp;` 放最后，避免二次解码。
- ❌ 把 `display:none` 的 `<li>` 当隐藏跳过 → ✅ 它们是要抓的正文章节。
- ❌ 诱饵检测写死「慎重地转动门把打开门」之类关键字 → ✅ 认 `class="<data-c>"`，
  关键字在正文里也会合法出现，会误判。
